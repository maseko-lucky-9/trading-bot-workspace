## Bridge Architecture

### HTTP Bridge (Chosen Implementation)
- Protocol: HTTP REST via FastAPI + uvicorn
- Server location: macOS M5 Pro host — bot/core/bridge/http_server.py
- Client location: UTM Windows 11 ARM VM running MT5 EA
- EA endpoint: http://192.168.64.1:8080 (UTM host-only adapter IP)
- MT5 EA uses WebRequest() with GET/POST — allow WebRequest in MT5 Tools → Options → Expert Advisors
- Heartbeat interval: EA POSTs heartbeat every 5 seconds; bot checks heartbeat.time on every tick read
- Heartbeat timeout: 10 seconds (config: bridge.heartbeat_timeout). Beyond this, treat bridge as disconnected.

### Key Endpoints
- POST /tick — EA pushes live tick data (bid, ask, spread, time, OHLCV H1)
- POST /account — EA pushes account state (balance, equity, margin)
- GET /command — EA polls for pending bot commands (orders, close, etc.)
- POST /result — EA confirms order execution results
- GET /health — liveness probe for bridge monitoring

### Known MT5 WebRequest Quirks
- WebRequest has a hardcoded 5s timeout in MQL5. Bridge must respond within 4s to avoid EA-side timeout.
- MT5 EA runs on the timer event (OnTimer) not on every tick. Set timer to 1000ms (1s) for H1 trading — not faster.
- If MT5 terminal is minimised, OnTimer fires at reduced frequency. Keep terminal visible during live trading.
- Allow DLL Imports must be checked in MT5 for ZMQ variant (not needed for HTTP/WebRequest).
- WebRequest requires whitelist entry for 192.168.64.1 in MT5 Tools → Options → Expert Advisors → "Allow WebRequest for listed URL".

### File-Based IPC (Legacy — do not use for new code)
- core/bridge/mt5_client.py reads/writes files from shared folder at /Volumes/mt5bridge
- Only retained for fallback testing without UTM running
- New code should use HTTP bridge client (US-001)

## Data Characteristics

### Primary Instruments
- EURUSD — primary, 5-digit quotes (0.00001 pip), 10 USD/pip/standard lot
- GBPUSD — secondary, same structure as EURUSD

### Timeframe
- H1 (60-minute candles) — primary trading timeframe
- Data granularity sufficient for EMA crossover (slow period up to 50 candles = ~50 hours lookback)

### Demo Account
- All initial development and paper trading uses a demo account
- Equity target for live mode enablement: > 1000 USD
- No slippage model needed for demo; paper broker uses mid-price fill

### Historical Data
- Requested via /history endpoint on bridge (EA calls MT5 CopyRates internally)
- Cache location: bridge_data/history/{symbol}_{timeframe}.parquet
- Cache TTL: 1 hour for H1, 15 minutes for M15 and below
- Minimum history required for indicators: max(slow_ema_period, atr_period) + 10 candles = ~64 candles minimum

## Target Performance Metrics (Autoresearch Guards)

These are the minimum acceptable thresholds. Any parameter set that fails ANY of these is rejected by the backtest guard:

- Sharpe ratio: > 1.5 (annualised, 252 trading days)
- Maximum drawdown: < 5% (peak-to-trough on equity curve)
- Win rate: > 45% (fraction of profitable closed trades)

Source: config.yaml autoresearch section. Do not modify these without updating config.yaml.

## Risk Framework

### Position Sizing
- Max risk per trade: 1% of equity (config: risk.max_risk_per_trade = 0.01)
- Kelly fraction: 0.25 (quarter Kelly — conservative start)
- Formula: size = (equity * 0.01) / (|entry - sl| / pip_value)
- After Kelly: size *= 0.25
- Round to 0.01 lots; minimum 0.01 lots

### Drawdown Thresholds (config: risk.*)
- 10% DD (trailing_dd_warn): log warning, reduce position size 30% (multiplier 0.7)
- 15% DD (trailing_dd_reduce): reduce position size 50% (multiplier 0.5)
- 20% DD (trailing_dd_halt): halt all trading (multiplier 0.0)
- Daily loss limit: 2% of equity (daily_loss_limit = 0.02) — hard stop for the day
